### [MEDIUM] jump-model-global-viterbi-unbounded-reach
- **Where:** modules/labeler.py:511-562 (generate_labels_jump_model), 467-485 (_jump_viterbi), 488-508 (_jump_fit)
- **Mechanism:** The labeler clusters feat=[fwd_ret[fk], fwd_vol[fk]] over sel = (~isnan(fwd_ret[fk])) & fv — this row set spans TRAIN, VAL and TEST. _jump_fit alternates a TRAIN-only centroid M-step with a Viterbi assignment over ALL rows, and the final labels come from _jump_viterbi(z, cents, lam) on the full z. With lam>0 (the sweep includes lam_mult=3.0) the Viterbi backtrace couples every index to every later index: states[-1] = argmin(dp) is set by the LAST (test) rows and propagates backward via back[i, states[i]] without any bound, and the M-step centroids are also indirectly shifted because cluster membership (st==k on TRAIN rows) depends on the global path. So a TRAIN or VAL label at any index t can be a function of TEST-period forward returns/vol — unlike sticky_hmm (lines 1016-1026) and kllt (lines 2999-3008), which block-decode at cut=first te_m index precisely to forbid this, jump_model has no cut. The declared 3rd return is best_h=fk (<=200), so the footer's _EMBARGO=max(200, horizons, label_reach) stays at 200 and is blind to the unbounded reach — the exact dc_reversal-class defect (993edcf) on a different labeler.
- **Reaches OOS results:** Two concrete paths in templates/footer.py.tmpl: (1) TRAIN labels contaminated by test-period fwd_ret/fwd_vol enter the XGBoost fit (tx = fv & ly & tr_m, line ~302), so the SAVED OOS test predictions themselves become functions of test-period forward returns; (2) VAL labels at positions earlier than tepos[0]-_EMBARGO pass the emb purge (line 295) and enter the isotonic calibrator fit and val_auc (vx_lab) even though they can encode test data — val_auc gates cell saving and best-cell selection, biasing which configuration's OOS Calmar is reported. jump_model is in LABELERS and FEATURED_LABELERS (lines 3542, 3562) so it competes in full sweeps; it is not a current book champion, so deployed book numbers are not affected today, but any future crown won by jump_model would carry inflated OOS.
- **Guard:** Apply the sticky_hmm/kllt block-decode precedent: compute cut = first te_m index; run _jump_fit/_jump_viterbi on z[:cut] only (TRAIN-fit centroids unchanged) and decode z[cut:] separately with the frozen centroids (or simply assign test rows by nearest-centroid, the lam=0 path). Also declare reach honestly until fixed. Regression test (generic, catches the whole class): for every labeler in LABELERS, generate labels twice — once on the full arrays, once with all te_m-period inputs (lc/lr/fwd_ret/fwd_vol beyond tepos[0]) replaced by a perturbed copy — and assert labels at indices < tepos[0] - max(200, declared_reach) are identical (append-OOS-invariance at the label level, mirroring tests/test_bar_threshold_leak.py).

### [LOW] visgraph-reach-off-by-one
- **Where:** modules/labeler.py:2839-2888 (generate_labels_visgraph, window at 2845, declared reach at 2885)
- **Mechanism:** The HVG window is `w = lc[i+1 : i+2+H]` — H+1 forward closes, indices i+1..i+H+1, so true reach is H+1; the 3rd return declares best_h = H. With the built-in horizons [40,80,160] the worst reach is 161 < the 200 embargo floor — safe. But the footer passes horizons=_hz=CONFIG['horizons'] when set (footer.py.tmpl:730,757-759), and _EMBARGO = max(200, max(horizons), declared H). For max(horizons)=200 the embargo is 200 while visgraph reaches 201: the VAL bar at position tepos[0]-201 passes the emb purge (emb = _allidx < tepos[0]-200) yet its label's window includes lc[tepos[0]] — the first TEST close enters the calibrator/val_auc fit set. - **Reaches OOS results:** Only under a horizons override with max >= 200 (the intraday _hz lever has historically used SHORT horizons, so this is latent today): exactly one fit-eligible VAL row's label encodes the first OOS close, entering cal.fit/eval_set and val_auc — a 1-bar, 1-row version of the dc_reversal boundary leak. Practical OOS distortion is negligible at current configs; flagged because the declared-reach contract (the thing _EMBARGO trusts) is violated by construction.
- **Guard:** Either shrink the window to `lc[i+1 : i+1+H]` (H closes) or declare reach H+1 (`return y, cfg, int(max(Ls-equivalent))+1`). Add the generic per-labeler empirical-reach regression test from the jump_model finding — it would catch this off-by-one automatically (perturb bar tepos[0], assert no label change at tepos[0]-201 for declared reach 200).

## Surfaces audited clean (finder summaries)

### bars-threshold-scope
modules/bar_builder.py fits one threshold (plus optional frozen params) per bar axis and then streams the full minute series through a causal O(1) builder. The leak lens was full-series statistics influencing thresholds/params.
